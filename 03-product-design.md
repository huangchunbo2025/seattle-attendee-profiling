### 2.3 完整公司情报卡 (交付物 B) — 11 张

**格式: 每家公司一个折叠面板(accordion)，展开后分四层信息深度**

**层级 1 — 速览区 (折叠状态即可见)**

| 字段 | 说明 | 示例 |
|---|---|---|
| 公司名 + 优先级标签 | P0/P1/P2/P3/Partner 色彩区分 | Canada Goose `P0` |
| 参会人 + 职位 | 姓名 / Title | Johnney Cao / Dir, D365 Solution Architecture |
| WHY NOW 一句话 | 为什么此刻有机会 | 新 CDIO + 招 VP AI + D365 核心 = 技术路线图重写中 |

**层级 2 — 对话准备区 (第一次展开)**

| 字段 | 说明 |
|---|---|
| HOOK — 开场破冰 | 2-3 个可选开场话题，含具体措辞建议 |
| PAIN — 核心痛点 | 最多 2 个，每个一句话 + 一段证据出处 |
| ASK — 期望下一步 | 一个具体、可衡量的行动(约 demo / 约会议 / 发资料) |
| RED LINE — 雷区 | 2-3 条绝对不能说的话 |

**层级 3 — 深度背景区 (GTM 团队用)**

| 字段 | 说明 |
|---|---|
| AI 近况 | 公司 AI 战略、投入、招聘信号、公开声明 |
| 会员/忠诚度现状 | Loyalty program 细节、CRM/CDP 技术栈、个性化成熟度 |
| MarTech 技术栈 | ESP、CDP、CRM、BI 供应商清单 + 集成关系图 |
| 竞争态势 | 现有供应商锁定程度、替换/增强可行性 |

**层级 4 — 参会者个人档案 (知己知彼)**

| 字段 | 说明 |
|---|---|
| 职业履历摘要 | 前 2-3 家公司 + 关键角色 |
| 决策影响力评估 | 在采购决策中的角色(决策者/强影响者/执行者) |
| 个人兴趣/共同背景 | 可用于破冰的非业务信息 |
| LinkedIn 链接 | 一键跳转 |

## 三、信息架构

### 3.1 设计原则: 30 秒法则

CEO 在活动现场的信息消费模式是 **"扫一眼 → 回去聊"**，不是 **"坐下来读文档"**。信息架构必须服务于这个行为模式。

### 3.2 导航结构 (HTML 单页)

```
┌─────────────────────────────────────────────┐
│  顶部固定导航栏                               │
│  [CEO 速查] [情报卡] [优先级] [Demo] [执行]    │
├─────────────────────────────────────────────┤
│                                              │
│  Section 1: CEO 速查卡                        │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐        │
│  │CG P0 │ │IHG P0│ │AT P1 │ │JC P1 │ ...    │
│  └──────┘ └──────┘ └──────┘ └──────┘        │
│  (水平滑动卡片，每屏一张)                       │
│                                              │
│  Section 2: 合作优先级矩阵                     │
│  [气泡图] [评分表] [切入路径]                   │
│  (三个 tab 切换)                              │
│                                              │
│  Section 3: 完整情报卡 (11 张)                 │
│  ▸ Canada Goose [P0]  ← 折叠面板              │
│  ▸ IHG [P0]                                  │
│  ▸ Arc'teryx [P1]                            │
│  ▸ JCPenney [P1]                             │
│  ...                                         │
│                                              │
│  Section 4: Demo 策略速查表                    │
│  (模块表格 + 展开详情)                         │
│                                              │
│  Section 5: 活动执行 Checklist                 │
│  (可勾选时间轴)                               │
│                                              │
└─────────────────────────────────────────────┘
```

### 3.3 快速定位机制

| 机制 | 实现 | 场景 |
|---|---|---|
| **顶部锚点导航** | 固定在屏幕顶部的 5 个按钮 | 在任何位置一键跳转到目标区域 |
| **公司名搜索** | 页面内搜索 (Ctrl+F / 浏览器内搜) | CEO 想快速找某家公司 |
| **优先级色彩编码** | P0=红, P1=橙, P2=蓝, P3=灰, Partner=绿 | 视觉扫描即可识别重要度 |
| **折叠/展开控制** | 默认折叠，点击展开详情 | 避免信息过载，按需钻取 |
| **CEO 速查卡独立置顶** | 永远是页面第一个 section | CEO 打开页面直接看到最需要的信息 |

### 3.4 阅读路径设计

**CEO 路径 (3 分钟)**:
1. 打开页面 → 自动停在 CEO 速查卡
2. 左右滑动 5 张卡片 → 记住 4 家重点公司
3. 如需更多细节 → 点击卡片跳转到完整情报卡
4. 活动中 → 手机打开直接看速查卡

**GTM 路径 (30 分钟)**:
1. 先看优先级矩阵 → 理解全局优先级和分工
2. 展开自己负责的公司情报卡 → 精读深度背景
3. 查看 Demo 速查表 → 确认技术演示准备
4. 打印 Checklist → 活动当天照着执行

## 四、交付形式

### 4.1 主交付: 响应式 HTML 单页

| 属性 | 选择 | 理由 |
|---|---|---|
| **格式** | 单个 HTML 文件 (自包含 CSS/JS) | 无需服务器，离线可用，用手机浏览器直接打开 |
| **响应式** | 手机优先设计，桌面端兼容 | CEO 活动现场用手机看，GTM 团队在笔记本上看 |
| **离线支持** | 所有内容内联，不依赖 CDN | 活动场地 WiFi 不可靠 |
| **可打印** | `@media print` 样式优化 | GTM 团队可打印 Checklist 和情报卡随身携带 |
| **无需安装** | 浏览器直接打开 | 零学习成本 |

### 4.2 为什么不选其他格式

| 格式 | 排除理由 |
|---|---|
| PDF | 无交互(折叠/展开/搜索)，手机端阅读体验差 |
| PPT/Slides | 线性阅读，无法快速跳转到目标公司 |
| Notion/飞书文档 | 依赖网络和登录，活动现场不可靠 |
| 打印卡片 | 11 张物理卡片容易弄乱，无法搜索 |
| 移动 App | 开发成本过高，4 天内无法交付 |

### 4.3 技术规格

- 单文件 HTML，体积控制在 500KB 以内
- 使用 CSS Grid/Flexbox 实现响应式布局
- 内联轻量 JS 实现折叠/展开、Tab 切换、卡片滑动
- 优先级色彩使用高对比度配色，确保手机屏幕阳光下可读
- 气泡图使用内联 SVG，不依赖外部图表库
- `@media print` 将所有折叠内容展开，确保打印完整

## 五、边界定义

### 5.1 MVP 范围 (5/22 活动前交付)

| 包含 | 说明 |
|---|---|

## 迭代记录

### Round 1: "挑剔用户" — CEO 在活动现场真的会看这些吗？

**挑战**: CEO 在社交场合不会掏出手机读文档。即使做成手机页面，他什么时候看？晚宴桌上掏手机很不礼貌。

**发现与修正**:

1. **使用时机重新定义**: CEO 看手机的窗口只有三个 — (a) 到场前在车上 (b) 去洗手间途中 (c) 活动间歇。速查卡必须在这些极短窗口内有用，所以从"30 秒可读"压缩到"10 秒可扫"。已将速查卡字段从 5 个精简为 4 个(痛点、开场、目标、红线)，每个不超过一行。

2. **真正的"活动前"准备才是核心**: CEO 的真实准备不是在活动现场看手机，而是在 D-1 晚上花 20 分钟记忆。速查卡的核心价值是"帮助记忆"，不是"现场查阅"。因此增加了"CEO 路径 3 分钟"的阅读路径设计，确保 D-1 晚上看一遍就能记住。

3. **Checklist 才是现场最有用的工具**: 相比情报卡，可勾选的 Checklist 对 GTM 团队在活动现场更实用(谁聊过了、谁还没覆盖)。已将 Checklist 设计为独立 section，可打印随身携带。

### Round 2: "业务方" — 对照 02-business-design.md，方案是否忠实反映业务意图？

**逐项对照**:

1. **业务目标 "3-5 个高质量 pipeline，1 个 90 天内 PoC"** — 产品方案的 Checklist 中包含了 48h/72h/1 周/2 周的跟进节点，与业务设计的 follow-up 计划完全对齐。通过。

2. **CEO 时间分配 60/30/10** — CEO 速查卡只做 P0+P1 共 5 张(4 家公司)，第 6 张汇总卡覆盖 P2/P3/Partner，体现了"其他人 GTM 处理"的原则。与业务设计对齐。通过。

3. **WFIT 四维评估模型** — 优先级矩阵直接复用了 WFIT 评分结果。MVP 中气泡图被降级为表格形式(Round 1 中确认 4 天内交付优先)，但评分逻辑完整保留。通过。

4. **三条切入路径 (Microsoft 生态/AI 增强/从零构建)** — 产品方案在优先级矩阵 section 中保留了切入路径视图，且在 Demo 策略中按路径关联了 Demo 模块。通过。

5. **座位安排策略** — 业务设计 Round 1 已修正为"不过度设计座位"，仅确保 CEO 身边坐 P0 参会者。产品方案的 Checklist 中体现为一条简单确认项，而非复杂座位图。通过。

6. **统一叙事框架** — 业务设计 Round 2 新增的 CEO 统一叙事，已作为独立 section 纳入产品方案，并在每张速查卡底部固定展示。通过。

**遗漏修正**: 业务设计中的"降级目标"(零 pipeline 产出时的保底成果)未在产品方案中体现。已在 Checklist 活动后 2 周部分增加 "全部 11 位确认进入 CRM"和"复盘"条目。

### Round 3: "工程师" — 方案可实现吗？有没有模糊需求？

**可实现性评估**:

1. **单文件 HTML 自包含**: 可行。所有交互(折叠/Tab/滑动)用原生 JS 实现，不需要框架。CSS 内联。预估开发 6-8 小时。

2. **手机滑动卡片**: 可行。CSS `scroll-snap-type: x mandatory` + `overflow-x: auto` 即可实现原生滑动体验，无需 Swiper 等库。

3. **折叠面板**: 可行。`<details>` + `<summary>` HTML 原生标签即可，零 JS 依赖。

4. **可勾选 Checklist**: 可行。`<input type="checkbox">` + `localStorage` 保存勾选状态，确保刷新后不丢失。

5. **打印优化**: 可行。`@media print` 中展开所有 `<details>`、隐藏导航栏、调整字体大小。

**模糊需求澄清**:

| 原始模糊点 | 澄清 |
|---|---|
| "气泡图可视化" | MVP 降级为表格，V1.1 再做 SVG 气泡图 |
| "高对比度配色" | 明确: P0=#DC2626(红), P1=#EA580C(橙), P2=#2563EB(蓝), P3=#6B7280(灰), Partner=#16A34A(绿) |
| "卡片滑动" | 明确: 使用 CSS scroll-snap，不依赖 JS 库 |
| "可打印" | 明确: Checklist 控制在 A4 双面 1 张; 完整情报卡每家 1 页约 11 页 |
| "500KB 体积限制" | 评估: 纯文本+内联 CSS/JS 约 80-120KB，远低于限制。如加 SVG 图表约 200KB。可行 |
| "localStorage 勾选状态" | 明确: 仅 Checklist 使用 localStorage，其他 section 无状态 |

**工程建议**: 建议先生成 Markdown 内容，再用单次 HTML 模板渲染。内容和样式分离开发，降低迭代成本。
