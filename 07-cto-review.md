## 三轮对抗迭代记录

### Round 1: 技术方案最薄弱的 3 个环节

#### R1-1: 数据模型存在危险的重复 — 严重

**问题**: `COMPANIES`、`CEO_CARDS`、`WFIT_SCORES` 三个数组都包含公司名、tier、参会者等重复字段。CEO 在 D-1 临时说"把 RealReal 从 P1-watch 升到 P1"时，开发者需要修改 3 处数据。遗漏任何一处就会导致速查卡和评分表显示不一致。

**修正**: 将 WFIT 评分内嵌到 `COMPANIES[i].wfit` 对象中。`tier` 只在 Company 上定义一次。CEO_CARDS 通过 `companyId` 引用 COMPANIES 数据，渲染时派生 tier 信息，不硬编码。增加 `calcWFIT()` 和 `getWFITScores()` 函数从 COMPANIES 派生评分表数据。

**影响范围**: 数据模型 Section 二 + JS 渲染逻辑 Section 九

#### R1-2: 邮件复制功能在目标环境不可用 — 中等

**问题**: `navigator.clipboard.writeText()` 需要 HTTPS 或 localhost 安全上下文。CEO 通过 `file://` 协议打开本地 HTML 文件时，Clipboard API 直接不可用。原方案的 fallback `document.execCommand('copy')` 已被标记为废弃，在 iOS Safari 上也不可靠。

**修正**: 放弃"一键复制"，改为"选中文本"策略。`selectTemplate()` 函数用 `Range` + `Selection` API 选中邮件模板文本，用户手动长按复制。这在所有浏览器和协议下都可靠。开发时间反而减少。

**影响范围**: 组件 S8 感谢邮件 Section 三

#### R1-3: 工时估算偏乐观，数据填充被严重低估 — 严重

**问题**: 原始估算 9.5h，优化后 7.5h，但"数据填充 2h"假设内容已经写好只需要格式化。实际情况是 11 家公司的深度信息需要从调研文档中提炼、整理为 JSON 结构，真实耗时 3-4 小时。

**修正**: 
- 数据填充改为 AI 辅助生成 JSON + 人工审核模式，压缩到 1.5h
- Phase A 中数据准备与骨架搭建并行执行
- 增加 D-2 交付检查点：5/20 晚必须完成代码，5/21 白天 CEO 实际手机验收
- 增加降级方案：如 5/20 晚未完成，砍搜索功能、打印优化、Checklist 持久化

**影响范围**: 开发计划 Section 十 Round 3

### Round 2: 生产事故回溯 — CEO 活动现场打开页面出问题

设定场景：CEO 在活动前 30 分钟打开手机浏览器查看速查卡，页面出问题，没有工程师在场。

#### R2-1: JS 执行失败导致页面空白 — 致命风险

**分析**: 单文件方案中，所有 CEO 速查卡和情报卡都由 JS 动态渲染。如果手机浏览器 JS 引擎出任何问题（内存不足、语法错误、iOS 版本兼容性），CEO 看到的就是一片空白加几个空壳 section。

**概率**: 中。虽然现代浏览器 JS 引擎很稳定，但 file:// 协议的行为比 HTTP 更不可预测。CEO 手机如果开了"低数据模式"或其他省电设置，可能影响 JS 执行。

**修正**: 增加 `<noscript>` 静态降级。7 张 CEO 速查卡以纯 HTML 形式写在 `<noscript>` 标签内。即使 JS 全挂，CEO 仍能看到文字版速查信息。实现成本 < 15 分钟。标记为 P0 必做项。

#### R2-2: 文件传输链路断裂 — 致命风险

**分析**: HTML 文件从开发者电脑到 CEO 手机的传输链路是最脆弱的环节。微信传文件会自动加 `.1` 后缀或改扩展名；邮件附件可能被企业邮箱安全策略拦截；iCloud Drive 同步可能延迟。

**修正**: 交付时提供 3 种传输方式确保至少一种成功。更关键的是，5/21 必须在 CEO 的实际手机上完成一次完整的打开和浏览测试。不是模拟器，不是另一台手机，是 CEO 本人的那台手机。

#### R2-3: iOS Safari 中英文混排渲染异常 — 中等风险

**分析**: 页面大量中英文混排（公司英文名 + 中文描述），iOS Safari 在某些字体回退场景下会出现行高跳变或文字截断。特别是在 375px 宽度 + 大字体的速查卡场景。

**修正**: CSS body 增加 `font-size: 16px`（防止 iOS 自动缩放到 12px 以下时的渲染异常）和 `-webkit-text-size-adjust: 100%`。开发阶段必须在真机 Safari 测试，不能只用 Chrome DevTools 模拟。

---

### Round 3: 开发效率优化

#### R3-1: Tab 切换组件是过度设计 — YAGNI

**分析**: WFIT 矩阵区域原设计为三 Tab 切换（评分表/切入路径/信号判读），需要 JS 状态管理和 CSS 切换动画。但实际使用场景是 GTM 团队从头到尾扫一遍，不会在 Tab 间反复切换。

**修正**: 去掉 Tab 组件，三个内容块纵向排列。节省 0.5h 开发时间，减少 JS 代码量约 30 行，降低出 bug 概率。去掉 `initTabs()` 函数。

#### R3-2: 列排序是过度设计 — YAGNI

**分析**: WFIT 表格的点击列头排序功能需要额外的 JS 逻辑和 UI 反馈。但表格只有 10 行，默认按总分降序已满足所有使用场景。没有人会在活动前一晚去按"Window 窗口期"重新排序。

**修正**: 去掉 `sortTable()` 函数，表格数据由 `getWFITScores()` 一次性按总分降序渲染。节省 0.5h。

#### R3-3: 开发阶段并行化

**修正**: Phase A 重新设计为两条并行路径：
- A1 路径：工程师搭建 HTML 骨架 + CSS 设计系统
- A2 路径：AI 辅助从调研文档生成 11 家公司 JSON 数据，人工审核

这两条路径无依赖关系，可以同时进行。Phase B 开始时，骨架和数据都已就绪，直接组装。

#### R3-4: 圆点指示器简化

**修正**: 圆点指示器保留滚动监听更新（必要的位置反馈），去掉点击跳转功能。scroll-snap 的手指滑动已经是最自然的切换方式，点击小圆点反而不直觉。节省 0.25h。

## 架构评价

### 做对了什么

1. **单文件 HTML 是唯一正确选择** — 离线约束 + 4 天交付 + CEO 零技术门槛，没有其他可行架构。
2. **数据驱动的 JS 渲染** — 虽然是单文件，内部数据-视图分离的思路正确，为 V1.1 迭代留了空间。
3. **原生 HTML 标签优先** — `<details>` 做折叠、CSS scroll-snap 做滑动、系统字体栈，零依赖是正确决策。
4. **localStorage + try/catch** — Checklist 持久化方案简洁可靠，隐私模式降级处理到位。
5. **打印优化单独设计** — 考虑到了 GTM 团队打印 Checklist 的真实场景。

### 需要警惕什么

1. **vanilla JS 生成大量 DOM 的可维护性** — 11 家公司 x 4 层信息 = 大量模板字符串拼接。没有组件化，没有类型检查。一个引号没闭合就全页崩溃。**建议**: 开发时使用模板函数隔离每个渲染单元，每写完一个函数立即在浏览器中验证。
2. **中文内容硬编码在 JS 字符串中** — 如果内容包含特殊字符（引号、反斜杠），会导致 JS 语法错误且难以调试。**建议**: JSON 数据部分先用独立工具验证 JSON 合法性，再粘贴到 HTML 中。
3. **没有自动化测试** — 单文件 HTML 无法做单元测试。唯一的质量保障是人工在真机上测试。这是可接受的（4 天项目不值得搭测试框架），但要确保 5/21 的验收测试覆盖所有关键路径。

## 长期建议

1. **V1.1 重构数据层**: 将 COMPANIES JSON 抽离为独立 `.json` 文件，HTML 通过 `fetch()` 加载。这需要 HTTP 服务（哪怕是 `python -m http.server`），但大幅提升数据维护体验。
2. **V2 考虑轻量框架**: 如果活动画像系统标准化为可复用模板（PRD V3 路线），单文件 vanilla JS 的可维护性会成为瓶颈。届时可引入 Alpine.js（14KB，单文件兼容）或 Preact（4KB）。
3. **建立活动画像模板库**: 本次手工填充 11 家公司数据的过程应被记录为 SOP，下次活动可用 AI 批量生成初稿、人工审核修正。
4. **CEO 手机型号固化测试**: 确认 CEO 使用的具体手机型号和 iOS 版本，将其作为唯一必须通过的测试环境。其他设备的兼容性在 MVP 阶段不重要。
